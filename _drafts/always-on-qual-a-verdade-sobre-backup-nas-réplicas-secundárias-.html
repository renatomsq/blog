Introdução

O seguinte documento tem como intuito validar quais backups podem ser realizados na réplica secundária, quais são as combinações possíveis e a explicação do porquê. Todos os testes foram realizados em ambiente de homologação. Não foi considerado nenhum teste utilizando ferramentas de backup (TeamVault ou Veeam), pois o intuito era validar apenas a relação de comportamento AG x Backup.

Validação de backups em réplicas secundárias

1 )Conectado no servidor primário, é criado um banco de dados chamado DBVALIDABACKUP.

  1. Ao tentar adicionar este banco no AlwaysOn AG, é solicitado um backup full, conforme evidência:

  1. Executar o seguinte comando para cumprir com a exigência do COPY_ONLY:

  1. Agora simulando a seguinte situação...se eu restaurar um banco do zero, ele ainda vai exigir um novo backup full para ingressar no AG? Ou se eu restaurar, o SQL Server entende que aquele backup de origem de restore de fato é o backup FULL exigido?

Ele deixa ingressar...Agora fazendo um teste final...e se mudarmos o modelo de recuperação de FULL para SIMPLE, pra depois voltar para FULL?

Ok, ele volta a reclamar e pede um backup full, mas e se eu realizar um backup diferencial?

O SQL Server entende que o backup diferencial caracteriza o banco que teve backup Full (critério explícito) e permite adicioná-lo novamente no ambiente.

Resumindo, a intenção de utilizar réplicas secundárias para realização de backup tem como objetivo desafogar a carga de trabalho de leitura e backup do ambiente primário, e foi constatado que, ainda será necessário realizar BACKUP FULL sem COPY_ONLY (devido ao diferencial) na primária, o backup diferencial também na primária (não há suporte para realização deste backup em réplica secundária), ficando o backup de log o único item que pode ser realizado em qualquer réplica sem quebrar cadeia de logs.

Para que a rotina de backup funcione desta forma, é necessário que a ferramenta que realize backup tenha inteligência de executar os backups FULL E DIFF na primária, enquanto o backup de log fica na secundária. É necessário implementar uma lógica parecida com esta:

sys.fn_hadr_is_primary_replica

Seria viável deixar todos os backups sendo realizados na réplica primária, se não houvesse  backup diferencial na rotina. Os backups diferenciais não são obrigatórios, tendo como intuito apenas acelerar o processo de recuperação, economizando a aplicação de restores de backup de log pois, o backup diferencial contém as modificações executadas desde o último backup FULL.

Extra:

Durante os testes, constatamos que as seguintes configurações são de caráter conceitual, pois independente de qual opção seja marcada, quem faz o backup que precisa implementar a lógica se vai preferir ou evitar réplicas secundárias.

As configurações só funcionam corretamente (e com muitas ressalvas) através de automated backups, termo cunhado que envolve backups realizados através de outro recurso do SQL Server, o Maintenance Plan.

Independente de qual opção esteja marcada,  fora backups de Maintenance Plan (recurso extremamente limitado e que não está em uso no ambiente da GEAP, as configurações não surtem nenhum efeito nos comandos disparados.

Realizei os testes também em ambiente de homologação, mas caso queira mais detalhes sugiro verificar o seguinte artigo, que explica no detalhe: https://www.sqlshack.com/understanding-backups-on-alwayson-availability-groups-part-1/

Resumo

  • Backups suportados em réplicas secundárias: FULL COPY ONLY e LOG;
  • Não é possível isolar backups na secundária se o backup FULL (sem COPY_ONLY) é necessário;
  • A ferramenta e/ou rotina precisa contar com uma lógica para identificar qual a primária/secundária para realizar corretamente a execução do backup;
  • Para deixar toda a rotina de backup na secundária (ou seja, FULL COPY ONLY E LOG) não é mais possível contar com backups diferenciais;
